# SELF IM - 2026-05-03

## 背景
本次在“箱码范围”需求上，前期出现多次理解偏差，导致实现路径反复、界面交互不聚焦，用户沟通成本升高。

## 主要问题
1. 先改代码后补全全局流程审计，导致局部优化偏离主目标。
2. 过度发散到额外按钮和流程，违背“简单易用、少设置”原则。
3. 对“末3位/末4位”语义理解不准确，没有第一时间绑定到箱码规则中的流水号。
4. 视觉与交互改动缺少一致性目标，出现可用性与美观度不足。

## 根因分析
1. 需求校准节奏不足：没有先固化“不可变约束”和“可变实现”。
2. 代码审计顺序不稳：未先完成入口-扫描-推测-预览全链路映射。
3. 反馈闭环滞后：未在每轮改动前用一句话复述关键业务口径并自检。

## 纠正动作（节能执行）
1. 先锁定约束再开发：每轮先写 3 行约束清单（入口名、自动/手动边界、数据口径）。
2. 先审计后改动：改 UI/流程前先检查现有页面状态机，避免重复控件和冲突按钮。
3. 单目标迭代：每次提交只完成一个主目标（功能正确性/交互优化/视觉优化三选一）。
4. 自动校验优先：先确保自动推测与自动匹配正确，再做展示层细节。
5. 完成前统一验证：运行测试并做关键路径自检后再提交，减少返工。

## 本次版本动作
- 版本号更新：`3.3.5 -> 3.3.6`
- 发布方式：测试通过后提交并推送 GitHub

## 后续执行标准
1. 默认“少按钮、少选择、低学习成本”。
2. 预览入口保持手动触发，范围结果保持自动更新。
3. 多图导入保持多选、多解析、批量结果可见。
